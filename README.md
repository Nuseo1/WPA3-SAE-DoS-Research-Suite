# WPA3-SAE DoS Research Suite

Welcome to the **WPA3-SAE DoS Research Suite**. This repository contains a powerful, Python-based framework designed to evaluate the robustness of WPA3 implementations against Denial-of-Service (DoS) attacks. It is directly based on the methodologies described in the 2022 academic paper: *"How is your Wi-Fi connection today? DoS attacks on WPA3-SAE"*.

⚠️ **DISCLAIMER:** This project is strictly for educational purposes and authorized security research. Only execute these scripts against networks you own or have explicit, documented permission to test. The authors are not responsible for any misuse.

***

## 📖 Overview

The WPA3-SAE (Simultaneous Authentication of Equals) handshake was designed to prevent offline dictionary attacks, but the complex cryptography involved in calculating the Password Element (PWE) makes the router's CPU and memory highly vulnerable to exhaustion attacks.

This suite contains two main tools:
1. **`sae_extractor.py`**: Automates the extraction of valid cryptographic payloads (Scalars & Finite Field Elements).
2. **`WPA3-SAE_DoS_Orchestrator_20_list.py`**: The main attack orchestrator that utilizes these payloads to simulate hundreds of unique, legitimate clients to bypass WIDS/WIPS systems.

***

## ⚡ The "20-List" Advantage
Early implementations of WPA3 DoS attacks sent the exact same cryptographic payload over and over again (single Scalar/Finite element). While these early DoS scripts sent identical payloads that simple WIPS/Firewall rules could easily fingerprint and drop before processing, this suite rotates 20 unique payloads to mimic a highly realistic, distributed burst of connection attempts.

Note: The WPA3 Anti-Clogging Mechanism (ACM) itself only relies on connection thresholds and token exchanges—it does not perform payload anomaly detection. However, rotating valid payloads ensures the attack traffic remains indistinguishable from legitimate devices, proactively bypassing potential heuristic WIDS/WIPS filters that monitor for static, repeated elliptic curve points.

This orchestrator implements a 20-List Rotation: By extracting 20 unique SAE parameter pairs (using 20 different fake passwords) and randomly rotating them for every single packet sent, the attack bypasses payload-fingerprinting. The router views every incoming packet as a completely new, legitimate connection attempt from a unique client, forcing it to allocate RAM and CPU cycles for each request, ultimately leading to device exhaustion.

## 🛠️ Prerequisites

- A Linux environment (Kali Linux or Ubuntu recommended)
- Python 3.x
- `scapy` library (`pip install scapy`)
- Root (`sudo`) privileges
- At least one Wi-Fi adapter supporting **Monitor Mode** and **Packet Injection** (e.g., ALFA Network AWUS036ACM). Two adapters are required to run the automated Extractor.

## 🚀 Step-by-Step Usage Guide

### Phase 1: Extracting SAE Parameters

To bypass WIDS, you must first extract real cryptographic payloads that match your target AP. The `sae_extractor.py` script automates this by spoofing connection attempts with random passwords and sniffing the resulting handshake.

1. **Configure `sae_extractor.py`:**
   Open the script and edit the `CONFIGURATION` block:
   ```python
   MANAGED_IFACE = "wlan1"     # Your adapter in Managed Mode
   MONITOR_IFACE = "wlan0mon"  # Your adapter in Monitor Mode
   TARGET_SSID = "Your_WiFi_Name"
   TARGET_BSSID = "AA:BB:CC:DD:EE:11"
   TARGET_CHANNEL = "11"
   ```

2. **Put your sniffing adapter into Monitor Mode:**
   ```bash
   sudo airmon-ng start wlan0
   ```

3. **Run the Extractor:**
   ```bash
   sudo python3 sae_extractor.py
   ```
   *The script will attempt to connect 20 times using random passwords. It will output two Python arrays containing 20 Scalars and 20 Finite Field Elements.*

### Phase 2: Configuring the Orchestrator

1. **Copy the extracted arrays:**
   Copy the `SAE_SCALAR_HEX_LIST` and `SAE_FINITE_HEX_LIST` generated by the extractor.

2. **Paste into the Orchestrator:**
   Open `WPA3-SAE_DoS_Orchestrator_20_list.py` and paste the arrays into the corresponding 2.4 GHz or 5 GHz configuration blocks.

3. **Set Target Variables:**
   In the orchestrator, configure the target MAC addresses and channel:
   ```python
   TARGET_BSSID_2_4GHZ = "AA:BB:CC:DD:EE:11"
   MANUELLER_KANAL_2_4GHZ = "11"
   ```

4. **Select your Attack:**
   Scroll down to `ADAPTER_KONFIGURATION` and assign your network adapter to your desired attack:
   ```python
   ADAPTER_KONFIGURATION = {
       "wlan0mon": {"band": "2.4GHz", "angriff": "double_decker"}
   }
   ```

### Phase 3: Launching the Attack

Once configured, run the orchestrator:

```bash
sudo python3 WPA3-SAE_DoS_Orchestrator_20_list.py
```

## ⚔️ Available Attacks

The Orchestrator implements 10 specific attacks categorized by their methodology:

### WPA3-Specific Attacks
*   **`omnivore`**: The strongest flooding attack. It cycles through random MAC addresses and SAE parameters, forcing the router to reserve RAM for every attempt until the memory is exhausted.
*   **`muted`**: Uses a single MAC address but rotates payloads. Useful against WIDS that only trigger on multi-source floods.
*   **`hasty`**: A state-machine confusion attack that sends SAE Commit and Confirm packets simultaneously to cause CPU spikes.
*   **`double_decker`**: A hybrid attack that starts with `omnivore` to bypass early defenses, followed immediately by a `muted` attack to maximize memory consumption after the Anti-Clogging Mechanism activates.
*   **`cookie_guzzler`**: Sends bursts of commits from random MACs to force the AP into sending disproportionately large SAE responses, clogging the channel.

### Cross-Band & Vendor Specific
*   **`radio_confusion`**: Targets Broadcom & MediaTek chips. If attacking from 2.4GHz, it targets the 5GHz BSSID (and vice-versa), crashing the opposite band due to shared memory architecture flaws.
*   **`back_to_the_future`**: Exploits a vulnerability where WPA2 APs crash when flooded with WPA3 authentication frames.
*   **`amplification`**: Spoofs the MAC addresses of legitimate clients, forcing the AP to respond to innocent devices, resulting in total channel saturation.
*   **`open_auth`**: Legacy DoS flooding the basic CPU queues.
*   **`deauth_flood`**: Standard IEEE 802.11 deauthentication attack.
